# Anhedral

[![npm version](https://img.shields.io/npm/v/anhedral.svg)](https://www.npmjs.com/package/anhedral)
[![node](https://img.shields.io/node/v/anhedral.svg)](https://www.npmjs.com/package/anhedral)

Anhedral is a deterministic, modular TypeScript application generator. It creates one pnpm/Turborepo workspace with only the app surfaces and backend features you select.

## Quick start

Run Anhedral in an empty directory with Node.js 20.19+ or 22.12+:

```sh
mkdir my-product
cd my-product
pnpm dlx anhedral@latest init
```

With no module flags, `init` selects the full stack. For a smaller project, pass modules explicitly:

```sh
pnpm dlx anhedral@latest init --web --api --db --auth
pnpm dlx anhedral@latest init --api
pnpm dlx anhedral@latest init --mobile --native-subscriptions
```

Generating without installation works on every supported generator runtime. Installing a workspace that includes Expo mobile requires Node.js 22.13+, 24.3+, or a newer release line; Anhedral checks this before changing the project and suggests `--skip-install` when the current runtime is incompatible.

Use `--skip-install` to generate files without resolving a workspace lockfile:

```sh
pnpm dlx anhedral@latest init --web --api --skip-install
pnpm install
```

## Modules

App surfaces:

- `web`: Next.js and shadcn/ui
- `mobile`: Expo Router
- `api`: Fastify
- `desktop`: Electron, Vite, React, and shadcn/ui
- `extension`: WXT, React, and shadcn/ui

Backend features:

- `db`: Neon and Drizzle
- `auth`: Clerk
- `billing`: RevenueCat webhook and entitlement persistence
- `storage`: Cloudflare R2 signed uploads
- `native-subscriptions`: RevenueCat native bootstrap for the Expo app (web target no-op)

Dependencies are resolved as a stable directed graph:

```text
auth                 -> api + db
billing              -> auth
storage              -> auth
native-subscriptions -> mobile + billing
```

## Commands

```text
anhedral init [modules...] [--toolchain <stable|latest>] [--skip-install] [--dry-run] [--json]
anhedral add <module...> [--skip-install] [--dry-run] [--json]
anhedral doctor [--json]
```

Add modules from an existing generated project:

```sh
pnpm dlx anhedral@latest add desktop extension
pnpm dlx anhedral@latest add storage --dry-run
```

`--dry-run` builds the complete plan in a temporary staging directory and leaves the project unchanged. If an interrupted transaction is pending, a dry-run refuses to recover it; rerun the original command without `--dry-run` first. `--json` emits one machine-readable JSON document without decorative CLI output; failures include a stable `code` alongside the human-readable `error`. `--verbose` streams child-command output for interactive diagnostics (JSON mode remains quiet so its output stays parseable).

The default `stable` toolchain uses verified, exact versions and deterministic local templates. `--toolchain latest` is intended for upstream drift testing.

A full selection currently produces this high-level structure:

```text

├── apps/
│   ├── api/
│   ├── desktop/
│   ├── extension/
│   ├── mobile/

├── packages/

│   └── db/
├── ANHEDRAL.md
```

The tree is conditional. For example, `--api` creates the Fastify app and contracts package, but no database, Clerk, storage, billing, or client packages. The source distribution maintains exact golden-tree contracts for representative stacks, including modes, sizes, and SHA-256 digests.

## Safe incremental generation

`anhedral.json` schema v3 records:

- the canonical resolved module closure;
- the exact generator version and selected toolchain channel;
- every generated file's owner, ownership class, SHA-256 hash, and portable mode.

Files are classified as managed, mergeable, or user-owned. `anhedral add` refuses to overwrite modified managed files, user-owned files, symlinks, or unowned path collisions. Root package fields and supported root configuration are merged structurally, while existing user workflows and documentation remain untouched. pnpm-only install and security policy lives in `pnpm-workspace.yaml`; safe-add preserves custom list entries and refuses conflicting generated mapping values.

Incremental generation is exact-current: `add` requires the manifest's generator version to match the running CLI. Regenerate projects created by another generator version instead of carrying compatibility code forward.

Generation uses a lock, a project-local reserved transaction namespace, an atomically written recovery journal, and intent-first backups. File contents, journal transitions, renames, and supported directory entries are synchronized before the transaction advances. A failed commit rolls the workspace back; a later invocation recovers an interrupted commit before doing new work.

## Local development in a generated project

```sh
pnpm install
cp apps/api/.env.example apps/api/.env
cp packages/db/.env.example packages/db/.env
cp apps/web/.env.example apps/web/.env.local
cp apps/mobile/.env.example apps/mobile/.env
cp apps/desktop/.env.example apps/desktop/.env
cp apps/extension/.env.example apps/extension/.env
pnpm db:generate
git add packages/db/migrations
pnpm verify
pnpm db:migrate
pnpm dev
```

Copy only the examples and run only the commands for modules selected in your project. For a database project, generate and review the initial SQL migration, stage it for version control, and then run verification before applying it. Runtime tools resolve environment files from the package where their command runs, so server variables belong in `apps/api/.env`, database CLI variables in `packages/db/.env`, and client-safe variables in the corresponding app directory. The root `.env.example` is an aggregate credential inventory, not the runtime source for package commands. Anhedral never generates a real `.env` file.

Surface-specific commands are generated only when the surface exists:

```sh
pnpm dev:web
pnpm dev:mobile
pnpm dev:api
pnpm dev:desktop
pnpm dev:extension

pnpm verify:web
pnpm verify:mobile
pnpm verify:api
pnpm verify:desktop
pnpm verify:extension
pnpm verify:db
```

Database projects also receive `db:generate`, `db:migrate`, `db:check`, and `db:studio` scripts. `verify:db` requires migration SQL, confirms it is tracked when run in a Git worktree, and runs Drizzle's migration-history check. Generated Linux CI additionally runs `db:generate` and rejects any tracked or untracked change under `packages/db/migrations`, so every schema change must include its reviewed SQL migration and metadata.

API projects include V8-backed Vitest coverage. `verify:api` and generated CI run the API tests with quantitative coverage floors, including the billing webhook and storage concurrency suites when those integrations are selected.

## Environment and providers

Only selected integrations add dependencies, code, tables, and environment keys. Depending on the module set, generated examples can include:

- API: `PORT`, `TRUST_PROXY_HOPS`, comma-separated `CORS_ORIGINS`, `ANHEDRAL_DEMO`
- Neon: `DATABASE_URL`
- Clerk server: `CLERK_PUBLISHABLE_KEY`, `CLERK_SECRET_KEY`
- Clerk clients: `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`, `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY`, `VITE_CLERK_PUBLISHABLE_KEY`
- Clerk extension networking: `VITE_CLERK_FRONTEND_API_URL` and optional `VITE_CLERK_SYNC_HOST`
- RevenueCat: server-only `RC_WEBHOOK_SECRET` and `RC_SECRET_API_KEY`, `RC_ENTITLEMENT_ID`, and client-safe `EXPO_PUBLIC_RC_*`
- R2: `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET`, `CRON_SECRET`

Never commit real `.env` files. Production startup fails when a selected provider is missing or malformed: database URLs must use `postgres:`/`postgresql:` with real credentials and a non-local host; Clerk keys must use their `pk_live_`/`sk_live_` forms; RevenueCat and cron secrets must be high entropy and non-placeholder; and R2 credentials and bucket names must match Cloudflare's issued formats. Browser `CORS_ORIGINS` must be listed explicitly as exact HTTPS origins in production; literal `null` remains supported for the desktop file origin. `ANHEDRAL_DEMO=true` is also rejected in production.
The API trusts no forwarding proxy by default. Set `TRUST_PROXY_HOPS` only when the deployment is behind a known reverse-proxy chain, and set it to that exact hop count.

When `auth` and `api` are selected together, generated web and mobile apps include hooks that bind Clerk session tokens to the shared API client. Web auth projects also include signed-out sign-in controls and a signed-in user menu. Native subscription projects initialize RevenueCat with the loaded Clerk user ID, synchronize sign-in and sign-out identity changes, expose a native paywall action, and intentionally skip SDK configuration on Expo web.

Billing webhooks are treated as retryable triggers rather than authoritative entitlement snapshots. The API reconciles the affected customer against RevenueCat's server API, handles transfer source revocation and destination reconciliation, honors grace-period expiration, and completes the durable webhook claim only after reconciliation succeeds. Use a dedicated server-only RevenueCat key with customer-read access and a separate high-entropy webhook authorization secret.

Generated Chrome extension manifests derive validated, deduplicated host permissions from `VITE_API_URL`, `VITE_CLERK_FRONTEND_API_URL`, and optional `VITE_CLERK_SYNC_HOST`. Configure those values in the extension's production environment; no manifest source edit is required.

## Deployment

The root `vercel.json` uses Vercel's current `services` schema. It declares `apps/api` and `apps/web` as independent service roots, then routes `/api/(.*)` to the API before the web catch-all. Vercel preserves the original request path, so Fastify mounts its routes under `/api` locally and in deployment. The generated web client uses same-origin `/api` by default in production so preview and custom domains stay deployment-relative; local client examples use `http://localhost:8787/api`, while mobile, desktop, and extension production URLs remain absolute. Storage projects also schedule the authenticated staging cleanup route once per day; set a strong `CRON_SECRET` in Vercel and configure an R2 lifecycle rule for `staging/` as a longer-lived backstop. Select the Services framework preset in Vercel before deploying.

Browser uploads to R2 presigned URLs also require a bucket CORS policy. Replace the example origin with every exact browser origin that may upload (origins contain no path or trailing slash), then verify the browser preflight and signed `PUT`:

```json
[
  {
    "AllowedOrigins": ["https://app.example.com"],
    "AllowedMethods": ["PUT"],
    "AllowedHeaders": ["Content-Type"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3600
  }
]
```

Use `apps/mobile` as the EAS project root. Run `pnpm desktop:build` for a current-host Electron artifact; use the desktop package's explicit `build:mac`, `build:win`, or `build:linux` script in matching platform CI. Create the Chrome extension archive with `pnpm extension:zip`.

## Package API

The published package exposes a side-effect-free ESM API from `dist/index.js`. Importing `anhedral` does not parse process arguments or execute the CLI. The executable entry point is isolated in `dist/bin.js` behind `bin/anhedral.js`.
